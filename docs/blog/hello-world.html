Ciao mondo!

by Valerio Versace | 16-10-2016 | Traduzioni: English Italiano

Siamo lieti di darvi il benvenuto sul blog di Soluzioni Futura. In questo articolo voglio darvi una panoramica sulle tecnologie che abbiamo scelto di utilizzare per realizzare questo sito e il relativo blog.

La nostra idea era quella di realizzare un sito web facile e rapido da modificare, che fosse pulito, chiaro ed estremamente veloce per gli utenti; volevamo anche che il sito potesse rappresentare una piccola vetrina delle tecnologie che utilizziamo.

Applicazione

Il sito è realizzato con un nostro fork di Pelican, un generatore di siti statici fatto in Python. Pagine e articoli sono composti da template di Jinja2 in cui viene iniettato del contenuto preso da dei file Markdown. Il sito è in realtà formato da quattro soluzioni di Pelican: una per il blog in italiano, una per il blog in inglese, una per il sito principale in italiano e una per il sito principale in inglese; queste vengono compilate separatamente.

Per il pre-processing del CSS utilizziamo Sass; il design del sito web è curato dai nostri collaboratori Niccolò Menozzi e Riccardo Zanutta, col quale ci congratuliamo per aver recentemente ottenuto il riconoscimento di Site of The Day su Awwwards.

Per la natura del sito web abbiamo deciso di stare leggeri col JavaScript: un uso pesante del rendering lato client per un sito in cui tutto l'HTML è precompilato avrebbe impattato negativamente sulla velocità di fruizione per gli utenti invece che positivamente.

Utilizziamo principalmente tracciati svg per le immagini; ottimizziamo dinamicamente quelle in formato non vettoriale. I font sono tutti serviti direttamente dal nostro server e sono disponibili in eot, svg, ttf, otf, woff e woff2, in modo da servire sempre ad ogni client quello più adatto; abbiamo favicon per i principali device mobile e non, inclusi i tile Metro di Windows 8, 10 e Windows Phone.

Implementiamo i tag Open Graph per la condivisione sui social e stiamo lavorando per supportare i Microdata secondo il protocollo di schema.org, così come i ruoli WAI-ARIA per l'accessibilità.

Infrastruttura

Il sito è hostato tutto con servizi di Amazon Web Services; nello specifico i file sono su un bucket di Amazon S3 e tutto il contenuto viene distribuito tramite CloudFront, la Content Delivery Network (o CDN). Facciamo caching di tutto il contenuto statico sia nel browser che a livello di CDN.

Tutto il sito è servito in HTTPS, con termination dell'SSL sulla CDN; il sito supporta inoltre connessioni IPv6 end-to-end e serve le richieste attraverso il protocollo HTTP2, quando supportato dal client.

A livello di distribuzioni, ne utilizziamo quattro: una per www.soluzionifutura.it, una per il dominio senza www, una per il blog.soluzionifutura.it e una per il dominio dei contenuti statici. Abbiamo scelto di ospitare i contenuti statici su un dominio separato principalmente per due motivi: gestire più agevolmente le logiche di caching e servire questi contenuti cookie-less. Tutte le volte infatti che fate una richiesta HTTP inviate automaticamente al web server tutti i cookie relativi a quel dominio; servendo i contenuti statici da un dominio separato non dovete inviare al web server tutti i cookie ad ogni richiesta per ogni file statico.

Il certificato SSL è fornito da Amazon Web Services Certificate Manager; l'accesso ai bucket di S3 su cui sono ospitate le pagine del sito e gli statici è consentito solo a chi accede tramite la CDN.

Deployment

Per modificare il sito web - a livello di applicativo o di contenuti - si lavora direttamente sulla soluzione di Git; la soluzione è ospitata su CodeCommit di Amazon Web Services. Pelican di default suggerisce come strumenti per il building, l'automation dei task e il deployment Make o Fabric; noi abbiamo preferito utilizzare Gulp per tutto.

Nello specifico, utilizziamo Gulp per fare concat e minifying di CSS e JS, resize e ottimizzazione delle immagini; inoltre, utilizziamo Gulp per gestire caching e invalidation delle risorse statiche. A questo proposito abbiamo realizzato un modulo di npm che abbiamo chiamato gulp-rev-recursive.

Questo modulo crea un hash a partire dal contenuto del file con cifratura SHA-256 e appende una porzione di questo hash al nome del file: se il nome del nuovo file è diverso rispetto al precedente, vuol dire che il file è cambiato, altrimenti non vi sono state effettuate modifiche.

Se il file è stato cambiato, il programma sostituisce ricorsivamente tutti i suoi riferimenti in tutti i file della soluzione, e di conseguenza esegue la stessa operazione coi file che modifica. In questo modo se abbiamo effettuato delle modifiche su un file statico salvato in cache, nel browser o nella CDN, il file sarà rinominato e i suoi riferimenti saranno modificati correttamente in tutto il sito web.

A questo punto Gulp fa uno zip di tutta la soluzione del sito compilata e la salva in una nostra repository di packages per il deployment specificando un environment per cui la soluzione è compilata (development, staging o production per esempio), e chi è autorizzato può utilizzare una nostra command line interface per effettuare il deployment della soluzione compilata nell'ambiente di esecuzione corrispondente.

Il risultato

Siamo soddisfatti del nostro nuovo sito web e abbiamo in mente tante nuove implementazioni da sviluppare nel corso del tempo e contenuti da proporre, sia riguardo ai servizi da noi proposti e alla nostra attività come azienda, sia articoli di approfondimento tecnico per gli addetti ai lavori. Per il momento vi lasciamo con un'immagine dei nostri risultati di PageSpeed e YSlow relativi alle performance del sito web.
